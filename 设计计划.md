## 必须实现的评分逻辑：

---

### ① Matchup Score（对线克制）

数据源：

```text
champion_matchup
```

逻辑：

* Ally vs Enemy
* 取 win_rate 平均
* 支持扩展（role / version）

---

### ② Synergy Score（双人组合）

数据源：

```text
champion_synergy
```

逻辑：

* Ally 内部两两组合
* 平均 win_rate

---

### ③ Team Synergy Score（整体阵容）

数据源：

```text
champion_team_synergy
```

逻辑：

* 精确匹配组合
* 未命中 → 默认 50

---

### ④ Destructive Score（阵容合理性）

逻辑：

* 检测阵容缺陷：

    * 无控制
    * 全脆皮
    * 缺前排
* 当前允许简化实现（规则模拟）

---

### ⑤ Final Score（最终评分）

```text
final_score =
  0.4 * matchup +
  0.2 * synergy +
  0.2 * team_synergy +
  0.2 * (100 - enemy_destructive)
```

---

### ⑥ 输出解释（必须）

生成：

```text
matchup_detail
synergy_detail
team_synergy_detail
recommendation
```

## 本次阅读遇到的困难 / 设计歧义
- 最明显的歧义是：`domain.dto` 里原本只有一个偏“名字型”的 `DraftAnalysisRequestsDTO`，但红色方最后一选原型需要的是“ban + 已选 + 需求位置”的 ID 型请求，因此我一开始难以判断是复用旧 DTO 还是新增专用 DTO。
- 第二个歧义是：返回对象同时存在 `DraftAnalysis`（PO）和 `DraftAnalysisVO`（VO），但当前 v0.4 的目标更偏向“先把分析链路跑通并落库”，所以这次先保持 `DraftAnalysis` 作为服务输出与持久化对象，后续再按接口层需要补 VO 映射。
- 结论：v0.4 采用 `LastPickAnalysisRequestDTO` 作为 controller / service 共享入参；原本的 `DraftAnalysisRequestsDTO` 语义已被迁移/重命名，不再作为最后一选主流程输入。

## V0.4.1 交互补充记录（2026-04-02）
- 将 `LastPickController` 的内嵌请求类上移到 `domain.dto.LastPickAnalysisRequestDTO`，统一 controller / service 的请求契约。
- `LastPickAnalysisService` 改为直接接收请求 DTO；`LastPickAnalysisServiceImpl` 补了空值、空列表、位置过滤、候选筛选与分项评分的原型实现。
- 原型评分仍然遵循“规则引擎优先”的路线：对线、协同、阵容整体、阵容威胁四类信息先跑通，后续再接大模型和 pgvector 解释层。
- 旧的名称型请求语义已从主流程中移出，后续若需要文本型/语义型分析再单独恢复，不要与最后一选 ID 型请求混用。

## V0.4.2 交互补充记录（2026-04-02）
- 前端开始进入原型阶段，采用 `frontend/` 作为独立目录，和 Spring Boot 后端并列，技术路线选择 Vue 3 + Vite + Element Plus。
- 首版前端页面不走“AI 风”装饰，改为暗色、信息密度较高的游戏工具面板风格，参考 OP.GG / 阿卡丽插件的布局感，但避免直接复刻具体视觉。
- 页面核心围绕“红色方最后一选”展开：输入 ban / ally / enemy / role，调用后端 `LastPickController`，展示最终推荐、分项评分和原始 JSON 解释。
- 当前前端默认请求路径需兼容后端现状：由于服务端同时存在 `server.servlet.context-path=/api` 与控制器 `/api/last-pick` 映射，联调时要注意接口前缀组合，后续如整理后端路由可再统一。
